# AVA Development Rules

These rules are persistent working rules for Codex on this project.

- Do not use Docker for AVA development, verification, or runtime work.
- Do not add new compose files, container runtimes, or desktop container dependencies unless the user explicitly reverses this rule.
- Do not start Docker Desktop during verification or use it as a hidden dependency.
- For services that need separate runtime processes, use already installed local/native services, Windows services, standalone binaries, or externally configured endpoints.
- AZOOM SFU/media must run as a native LiveKit executable, Windows service, or external server. Keep `SpringBoot/LiveKit/` runtime artifacts out of git.
- AVA/AZOOM must work from computers on different internet networks, not only the local LAN. Build packages and runtime config must use the public backend/media addresses, and AZOOM media must advertise public ICE candidates.
- AZOOM no longer provides text chat channels. Keep normal messenger chat in the `채팅` product, and keep AZOOM focused on voice channels, Notiva AI, and meeting transcripts.
- When implementing a requested change, identify the exact requested UI/logic surface and keep every unrelated UI and behavior untouched. Do not refactor, redesign, or adjust neighboring features unless the user explicitly asks for that specific area.
- GitHub `main` is the single source of truth shared by the Mac build machine, the Windows/Android build machine, and the Spring Boot server machine. Every machine must pull from `main` before work, push intended source/config/doc/test changes after work, and never keep private source edits that other machines need to build or run AVA.
- No development, implementation, build preparation, version bump, or release packaging may start until the machine has checked GitHub for newer commits. Always run `git status --short`, protect any local changes, run `git fetch origin`, compare local `main` with `origin/main`, and pull/rebase remote updates before changing files when the worktree is clean.
- Cross-machine sharing must happen through Git for source, platform project files, scripts, configuration templates, docs, tests, and generated source files that are intentionally part of the project. Do not rely on manual copy/paste, chat transcripts, local-only folders, or one machine's unpushed state for files required by Windows, Android, macOS, iOS, or the server.
- Build outputs and local runtime artifacts are not shared through Git: keep `build/`, `.dart_tool/`, `.gradle/`, `Pods/`, `DerivedData`, `dist/`, update packages, installer images, APK/IPA/DMG/ZIP files, logs, caches, and temporary server data ignored unless a rule explicitly says a small metadata/doc file must be tracked.
- Local secrets are never shared through Git: signing keys, keystores, provisioning profiles, Apple certificates, passwords, tokens, `.env` files, machine-local paths, and private service credentials must remain ignored and local to the machine/account that owns them.
- Platform ownership is explicit: Mac handles macOS/iOS builds and Apple signing handoff; the Windows/Android build machine handles Windows update ZIPs and Android release APKs with the AVA release signing key; the server machine pulls backend/client config changes, hosts `SpringBoot/AppUpdates` packages locally, and restarts Spring Boot after update manifest/package changes.
- Only require a Flutter app update when the implemented change changes client code/assets/platform projects, client-visible bundled behavior, client/server contracts that old clients cannot safely use, release update behavior, or any config that is compiled into the distributed app. Server-only fixes that existing released clients can consume through the current API/WebSocket contract must not bump the Flutter version or advertise a new app update; deploy/restart the server and verify the server-side behavior instead.
- When an app update is required, use `Flutter/tooling/bump_version.sh` on macOS or `Flutter/tooling/bump_version.ps1` / the matching `.cmd` wrapper on Windows unless a newer explicit version was already chosen. Do not report a required client update as complete while `Flutter/pubspec.yaml` and `Flutter/lib/src/config/app_version.dart` still show the previous version.
- Version bump means updating all tracked version sources that represent the client release, including `Flutter/pubspec.yaml`, `Flutter/lib/src/config/app_version.dart`, backend update defaults in `SpringBoot/src/main/resources/application.yml` only when the manifests should advertise the new app version, and any matching docs/scripts that hard-code the release version.
- After a required version bump, create or explicitly hand off every matching app deliverable for the platforms being shipped: Windows update ZIP, Android release APK, macOS DMG, and iOS IPA when Apple signing is available. If the current machine cannot produce a platform artifact, commit/push the source change and state the exact remaining build/signing task for the owning machine.
- Do not move a backend latest update manifest to a new version unless an app update is actually required and the matching package for that platform exists on the server machine under `SpringBoot/AppUpdates` or the server's configured `AVA_APP_UPDATE_DIR`. After package placement, restart Spring Boot and verify the previous version reports `updateAvailable: true` for each shipped platform.
- The Android release APK must be built from the same Flutter version and copied/named as `Flutter/build/app/outputs/flutter-apk/ava-android-{version}.apk`.
- Android release APKs must always be built with the AVA release signing config from `Flutter/android/key.properties`; never ship an APK signed with the Android debug certificate.
- Android release signing files such as `Flutter/android/key.properties` and `Flutter/android/app/ava-release.jks` are local secrets and must stay ignored by git.
- Do not add `REQUEST_INSTALL_PACKAGES` back to the Android manifest for AVA mobile updates. Mobile updates download the APK to `Downloads/AVA`, then the user installs it from the downloaded file.
- Mobile updates are mandatory release behavior: every Android app build must keep the startup/resume update check enabled and must point to the Android update manifest/APK, just like Windows points to the Windows update package.
- After an Android release APK is shipped, do not replace or delete the local Android release signing key; future APK updates must be signed with the same certificate.
- Windows update packages, Android APKs, macOS DMGs, and iOS IPAs must be built against the public backend URL `http://112.166.136.198:8080` and WebSocket URL `ws://112.166.136.198:8080/ws`, not `localhost`, so other PCs and phones can connect.
- macOS update packages are local-install DMGs unless Apple Developer signing/notarization is explicitly completed. Keep DMGs out of Git, copy them to the server update directory for hosting, and verify `/api/app-updates/macos/latest?currentVersion=<previous-version>` plus the DMG download endpoint.
- On macOS, use `Flutter/tooling/package_macos_update.sh` after the version bump to build the release app with the public API/WS URLs, create `AVA_Project_<version>_<build>_macOS.dmg`, and copy it into `SpringBoot/AppUpdates`. Only then may the macOS backend update defaults or server environment variables advertise that version.
- iOS update packages require Apple Developer signing/provisioning. Do not fake or bypass signing; when signing is not available, commit/push the source and leave the IPA task as an explicit manual Apple-account step.
- On Windows machines where Flutter symlink creation is unavailable, desktop run/build scripts must run `flutter pub get`, then `Flutter/tooling/ensure_windows_plugin_junctions.ps1`, then Flutter with `--no-pub`. Do not call raw `flutter run -d windows` or raw `flutter build windows` in that environment because it can delete plugin junctions and fail before the desktop app opens.
- Before starting any implementation or modification, sync from GitHub first: check `git status --short`, protect any existing user/local changes, then run `git fetch origin` and `git pull --rebase origin main` only when the worktree is clean or after the existing changes are safely committed/stashed with explicit intent. Never overwrite, reset, delete, or rebase away uncommitted user work.
- After every implementation or modification, commit the matching project/version changes, pull/rebase from GitHub, push to GitHub, and verify `git status --short` is empty so VS Code Source Control is empty before reporting completion.
- If VS Code Source Control is not empty, clear it without damaging `AVA_PROJECT`: commit intended source/config/test/documentation changes, keep local secrets and runtime build artifacts ignored, and never use destructive reset/checkout/delete operations against user work.
- Commit messages must describe the actual shared change, not the local machine action. Prefer messages like `Enable Apple app update prompts`, `Bump AVA to 0.1.296`, or `Add macOS update manifest support`; avoid committing unrelated local cleanup together with release/version work.
- When multiple machines are collaborating on one release, use this order: `git status`, `git fetch`, `git pull --rebase origin main`, make scoped changes, run the relevant platform checks, `git diff`, commit, `git pull --rebase origin main`, push, then tell the next machine exactly which commit/version/artifact to pull or build.
