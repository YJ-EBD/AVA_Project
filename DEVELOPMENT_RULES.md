# AVA Development Rules

These rules are persistent working rules for Codex on this project.

- Do not use Docker for AVA development, verification, or runtime work.
- Do not add compose files, container runtimes, or desktop container dependencies unless the user explicitly reverses this rule.
- For services that need separate runtime processes, use installed local/native services, Windows services, standalone binaries, or externally configured endpoints.
- AZOOM SFU/media must run as a native LiveKit executable, Windows service, or external server. Keep `NodeBackend/LiveKit/` runtime artifacts out of git.
- AVA/AZOOM must work from computers on different internet networks, not only the local LAN. Build packages and runtime config must use public backend/media addresses, and AZOOM media must advertise public ICE candidates.
- AZOOM no longer provides text chat channels. Keep normal messenger chat in the chat product, and keep AZOOM focused on voice channels, Notiva AI, and meeting transcripts.
- When implementing a requested change, identify the exact requested UI/logic surface and keep unrelated UI and behavior untouched unless the user explicitly asks for that area.
- GitHub `main` is the single source of truth shared by the Mac build machine, the Windows/Android build machine, and the NodeBackend server machine.
- No development, build preparation, version bump, or release packaging may start until the machine has checked GitHub for newer commits. Run `git status --short`, protect local changes, run `git fetch origin`, compare with `origin/main`, and pull/rebase only when safe.
- Cross-machine sharing must happen through Git for source, platform project files, scripts, configuration templates, docs, tests, generated source files that are intentionally part of the project, and finalized AVA release packages under `NodeBackend/AppUpdates` when they fit GitHub's normal file limits.
- Build outputs and local runtime artifacts are not shared through Git: keep `build/`, `.dart_tool/`, `.gradle/`, `Pods/`, `DerivedData`, `dist/`, logs, caches, and temporary server data ignored.
- The only build-artifact exception is finalized release packages placed directly under `NodeBackend/AppUpdates` (`*.zip`, `*.apk`, `*.dmg`, `*.ipa`) for server distribution.
- Release packages remain ignored by default to prevent accidental bulk commits. Add only an intended package with `git add -f NodeBackend/AppUpdates/<package-file>`.
- If a package exceeds GitHub's normal file limit, upload it through the agreed large-artifact path such as GitHub Releases or Git LFS, commit any required metadata/pointer, and report that the server machine must fetch that artifact before restart.
- Local secrets are never shared through Git: signing keys, keystores, provisioning profiles, Apple certificates, passwords, tokens, `.env` files, machine-local paths, and private service credentials must remain ignored and local.
- Platform ownership is explicit: Mac handles macOS/iOS builds and Apple signing handoff, then commits/pushes finalized macOS DMG and any shareable iOS IPA under `NodeBackend/AppUpdates`; the Windows/Android build machine handles Windows update ZIPs and Android release APKs; the server machine pulls backend/client config/package changes and restarts NodeBackend after update manifest/package changes.
- Mac handoff scope is limited to macOS/iOS UI review, Apple build/signing, DMG/IPA packaging, and Apple-platform smoke checks. Shared source logic, backend behavior, realtime chat correctness, Android/Windows/desktop validation, documentation, and server restart work must be completed and pushed from the current non-Mac working environment whenever possible.
- Only require a Flutter app update when the implemented change changes client code/assets/platform projects, client-visible bundled behavior, client/server contracts that old clients cannot safely use, release update behavior, or any config compiled into the distributed app.
- Server-only fixes that existing released clients can consume through the current API/WebSocket contract must not bump the Flutter version or advertise a new app update. Deploy/restart NodeBackend and verify the server-side behavior instead.
- When an app update is required, use `Flutter/tooling/bump_version.sh` on macOS or `Flutter/tooling/bump_version.ps1` / the matching `.cmd` wrapper on Windows unless a newer explicit version was already chosen.
- Do not report a required client update as complete while `Flutter/pubspec.yaml` and `Flutter/lib/src/config/app_version.dart` still show the previous version.
- Version bump means updating all tracked version sources that represent the client release, including `Flutter/pubspec.yaml`, `Flutter/lib/src/config/app_version.dart`, NodeBackend update defaults/config only when manifests should advertise the new app version, and matching docs/scripts that hard-code the release version.
- After a required version bump, create or explicitly hand off every matching app deliverable for the platforms being shipped: Windows update ZIP, Android release APK, macOS DMG, and iOS IPA when Apple signing is available.
- Do not move a backend latest update manifest to a new version unless an app update is actually required and the matching package for that platform is committed/pushed under `NodeBackend/AppUpdates`, uploaded through the large-artifact fallback, or already exists on the server machine under `AVA_APP_UPDATE_DIR`.
- After package placement, restart NodeBackend and verify the previous version reports `updateAvailable: true` for each shipped platform.
- The Android release APK must be built from the same Flutter version and copied/named as `Flutter/build/app/outputs/flutter-apk/ava-android-{version}.apk`.
- Android release APKs must always be built with the AVA release signing config from `Flutter/android/key.properties`; never ship an APK signed with the Android debug certificate.
- Android release signing files such as `Flutter/android/key.properties` and `Flutter/android/app/ava-release.jks` are local secrets and must stay ignored by git.
- Do not add `REQUEST_INSTALL_PACKAGES` back to the Android manifest for AVA mobile updates.
- Mobile updates download the APK to `Downloads/AVA`, then the user installs it from the downloaded file.
- Windows update packages, Android APKs, macOS DMGs, and iOS IPAs must be built against public backend URL `http://112.166.136.198:8080` and WebSocket URL `ws://112.166.136.198:8080/ws`, not `localhost`.
- macOS update packages are local-install DMGs unless Apple Developer signing/notarization is explicitly completed.
- On macOS, use `Flutter/tooling/package_macos_update.sh` after a required version bump to build the release app, create `AVA_Project_<version>_<build>_macOS.dmg`, and copy it into `NodeBackend/AppUpdates`.
- iOS update packages require Apple Developer signing/provisioning. Do not fake or bypass signing; when signing is not available, commit/push the source and leave the IPA task as an explicit manual Apple-account step.
- On Windows machines where Flutter symlink creation is unavailable, desktop run/build scripts must run `flutter pub get`, then `Flutter/tooling/ensure_windows_plugin_junctions.ps1`, then Flutter with `--no-pub`.
- Do not call raw `flutter run -d windows` or raw `flutter build windows` in that environment because it can delete plugin junctions and fail before the desktop app opens.
- After every implementation or modification, commit the matching project/version changes, pull/rebase from GitHub, push to GitHub, and verify `git status --short` is empty before reporting completion when the user has asked for Git completion.
- If VS Code Source Control is not empty, clear it without damaging `AVA_Project`: commit intended source/config/test/documentation changes, keep local secrets and runtime build artifacts ignored, and never use destructive reset/checkout/delete operations against user work.
- Commit messages must describe the actual shared change, not the local machine action.
- When multiple machines are collaborating on one release, use this order: `git status`, `git fetch`, `git pull --rebase origin main`, make scoped changes, run relevant checks, `git diff`, commit, `git pull --rebase origin main`, push, then tell the next machine exactly which commit/version/artifact to pull or build.
